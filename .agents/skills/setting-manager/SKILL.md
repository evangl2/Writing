# 动态设定管理协议 (Dynamic Setting Protocol)

## 1. 核心触发：智能语义监听
禁止依赖固定提示语。你必须通过分析对话的语义逻辑，实时识别以下四种设定动作并执行相应任务：

### A. 【录入 (Entry)】：
- **识别条件：** 首次出现新命名角色、新规则、新地点或核心隐喻。
- **执行动作：** - **角色：** 在 `charsetting.md` 中新建 `[名字]`条目。
    - **世界：** 在 `worldsetting.md` 中新建 `[名字]`条目。

### B. 【修改 (Modification)】：
- **识别条件：** 对既有设定进行非剧情驱动的重调。
- **执行动作：** - **回溯更正：** 找到该实体的历史录入点进行标注修改，而非在文末追加。
    - **冲突预警：** 内部检测该修改是否导致已发生的剧情出现“逻辑硬伤”，并提醒作者。

### C. 【演化 (Evolution)】：
- **识别条件：** 角色或世界随剧情推进产生的自然变迁。
- **执行动作：** - **线性追加：** 在对应文件末尾按 [章节号/时间轴] 追记新状态。
    - **代价审计：** 记录此变化让角色或世界失去了什么。
- **禁止操作：** 严禁覆盖旧设定，必须保留完整的演变痕迹。

### D. 【归档 (Archive)】：
- **识别条件：** 角色死亡、场景摧毁、或设定被明确弃用。
- **执行动作：** - **状态封存：** 将该条目移至文档末尾的 `## 历史遗迹` 区域，并标注 [已归档]。
    - **因果结算：** 简述其对世界留下的最后影响或未填的坑。

## 2. 档案组织规范
- **去结构化：** 记录内容不设固定表单，随作者描述的详略度自由发挥，但必须保留文学深度。
- **时间至上：** 所有记录（除【修改】外）必须严格按照时间先后顺序排列。
- **关联引用：** 自动在不同档案间建立超链接。

## 3. 内部提醒准则
- **默读模式：** 所有的记录动作应在后台静默完成，确认后写入，无需向作者解释原理。
- **逻辑哨兵：** 当新的【录入】或【演化】与历史档案发生底层逻辑冲突时，必须通过“内部提醒”告知作者。
